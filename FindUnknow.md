-
與 Claude Fable 5 協作的過程中，不斷讓我重新溫習一個老道理：「地圖不等於真實的疆域」。
地圖是工作內容的一種呈現形式，也就是我提供給 Claude 的提示詞（Prompts）、技能與上下文；疆域則是工作實際發生之處，也就是程式庫、真實世界以及其背後的實體限制。
地圖與疆域之間的落差，就是我所說的「未知」（Unknowns）。當 Claude 遇到未知時，它就必須根據對我意圖的最優猜測來做出決定。當要執行的工作量越大，Claude 可能碰到的未知就越多。
Fable 是第一個讓我感覺到「工作品質的瓶頸，竟然取決於我澄清未知的能力」的模型。
最重要的一點是，光靠提前規劃並不總是足夠的。你可能會在深入執行階段時才發現隱藏的未知，或者這些未知會直接點醒你：其實你應該用完全不同的方式來解決這個問題。
我發現與 Fable 協作是一個在執行前、中、後不斷挖掘未知的迭代過程。
我在這裡列舉了一些用來尋找未知的產出範例（Artifacts），但請記得隨時回來閱讀，以培養何時該使用它們的直覺。

▍了解你的未知
你的未知是什麼？當我帶著問題來找 Claude 時，我通常會將其拆解為四個維度：

已知圈的已知（Known Knowns）：這基本上就是我寫在提示詞裡的東西。我明確告訴 Agent 我想要什麼？

已知圈的未知（Known Unknowns）：那些我還沒想透，但我意識到自己還不知道的事情。

未知圈的已知（Unknown Knowns）：那些對我來說理所當然到根本不會寫下來，但只要一看到就能立刻認出來的事情。

未知圈的未知（Unknown Unknowns）：那些我壓根兒沒想過的事。我缺乏了哪些自己都不知道的知識？我是否知道某個東西能做到多完美的程度？

頂尖的 AI 程式開發者之所以厲害，是因為他們的未知相對很少。看著像 Boris 或 Jarred 這樣的高手寫提示詞，你會很清楚他們非常具體地知道自己要什麼。他們與程式庫以及模型的行為模式都處於高度同步的狀態。

但他們同樣會去假設未知的存在。在許多方面，減少並為你的未知做規劃，正是 Agent 協作編程的核心技能。幸運的是，這項技能你可以透過與 Claude 持續協作來不斷提升。

▍幫 Claude 幫到忙
引導 Claude 是一門細緻的平衡藝術。如果你限制得太死，即便中途需要轉向，Claude 依然會死板地執行你的指令；如果你給得太模糊，Claude 通常會基於業界的普遍最佳實踐來做決定與假設，而這些做法未必適合你的具體任務。

當你沒有把未知納入考量時，你兩頭都會落空：你不知道何時前方的道路會充滿障礙，你也不知道何時前方的道路其實一馬平川，但你卻依然希望 Claude 轉向。

Claude 可以幫你更快發現你的未知。它能極其迅速地檢索你的程式庫和網路，而且它在一般知識領域懂得比你多得多。同時，它從失敗中迭代的速度也更快。

這個過程中最重要的部分，是讓 Claude 了解你的起點。例如，告訴它你目前思考到哪裡了、交代你對這個問題或程式庫的熟悉程度，讓它扮演你的思維夥伴與你並肩作戰。

我先前寫過關於在 Claude 中使用 HTML 的文章，在接下來提到的大多數情況下，HTML 產出物（Artifact）都是拿來視覺化和呈現這些內容的最佳方式。

在本文中，我詳細列出了我用來揭示這些未知的幾種模式。我不會每次都用上所有技巧，但這是一套非常實用的工具箱。

▍執行前階段：盲點檢視（Blind Spot Pass）
在開始工作時，最實用的事情之一就是了解自己的盲點。例如，如果你要在程式庫中一個全新的區塊開發新功能，或是利用 Claude 來協助你不熟悉的工作（如迭代設計），你很可能會遇到大量的未知圈的未知。

你可能不知道該問什麼問題、不知道什麼樣才叫做好、不知道過去做過哪些歷史工作，或是不知該避開哪些坑。

這時，你可以直接請 Claude 幫你找出這些未知圈的未知並向你解釋。我喜歡直接使用「盲點檢視」和「未知圈的未知」這兩個詞。提供你一開始的身分與背景知識，對這個步驟通常至關重要。
提示詞範例：

「我正在著手新增一個身份驗證提供商，但我對這個程式庫中的驗證模組一無所知。你能幫我做一次盲點檢視，幫我找出相關的未知圈的未知，並指導我該如何更好地對你發問嗎？」

「我不知道什麼是調色，但我需要為這支影片調色。你能教我理解關於調色的未知圈的未知嗎？這樣我才能更好地向你提問。」

▍執行前階段：腦力激盪與原型設計（Brainstorms and prototypes）
當我工作的領域存在大量的未知圈的已知（即那些我講不出來、但一看到設計就能評斷的標準）時，我喜歡讓 Claude 和我一起進行腦力激盪與原型設計。

在原型設計階段儘早發現並口語化這些未知圈的已知是極具價值的，因為如果在執行階段才發現，代價通常會相對昂貴。功能或規格上的微小變動，都可能導致程式碼實作上的巨大差異，而且讓你的 Agent 去復原之前的修改也會變得更困難。

舉例來說，你可能只想看看在某個框架裡加個按鈕長什麼樣子，而不想大費周章去串接後端路由，或是在前端維護額外的狀態。
對我而言，視覺設計很難用言語精確表達，但我一看到畫面就知道是不是我要的。在這種情況下，我會要求對同一個產出物提供幾種不同的設計方向。

此外，我幾乎每次寫程式前，都會先進入探索或腦力激盪階段。這有助於我帶著清晰的意圖去定義專案的範疇。Claude 經常能發現我可能會忽略的高價值方法，有時它也會見樹不見林，而腦力激盪正好可以防止我把專案範圍訂得太窄或太寬。

提示詞範例：
「我想為這些資料做一個儀表板，但我毫無視覺美感，也不知道能做出什麼效果。請幫我生成一個 HTML 頁面，裡面包含 4 種截然不同的設計方向，好讓我能進行挑選與反饋。」

「在串接任何功能之前，請先製作一個單一的 HTML 檔案，用假資料模擬新的編輯器工具列。我想先對版面配置做評估，之後你再動到真正的應用程式。」

「這是我們大概遇到的問題：使用者在引導流程後就流失了。請檢索程式庫並腦力激盪出 10 個我們可以介入改善的地方，從成本最低到最具野心的方案排開。我會告訴你哪些方案引起我的共鳴。」

▍執行前階段：訪談（Interviews）
當做完足夠的腦力激盪後，我可能還是會存有一些未知。
在這種情況下，我會請 Claude 針對任何模糊或未知的地方對我進行訪談。在請 Claude 訪談你時，試著提供它關於問題的背景上下文，來引導它的提問。

提示詞範例：
「請一次問我一個問題，針對任何模糊不清的地方對我進行訪談。請優先提出那些如果我的答案不同，就會改變系統架構的問題。」

▍執行前階段：參考資料（References）
有時候你無法詳細描述你想要的東西。可能是你缺乏相關的專業術語，或者是它太複雜了，光是解釋就要花上你大半天。

這種時候，最佳解就是提供參考資料。雖然你可以附上圖表、文件或圖片，但最頂級的參考資料永遠是原始碼。

如果你發現某個套件實現了你想要的某種行為，或者某個設計組件深得你心，直接把 Fable 指向那個資料夾並告訴它要注意什麼，哪怕它是用完全不同的程式語言寫的也沒關係。

這也是 Claude Design 的工作原理。你不需要非得親手塞給它一個檔案。你可以直接把它指向你喜歡的網站上的某個模組，它讀取的是底層的程式碼，而不僅僅是一張截圖。這能提供關於標記、結構以及組件實際構建方式的豐富細節。

提示詞範例：
「這個位在 vendor/rate-limiter 的 Rust crate 實現了我完全想要的退避行為。請閱讀它，並在我們的 TypeScript API 客戶端中重新實作相同的邏輯語義。」

▍執行前階段：實作計畫（Implementation Plans）
當我認為一切就緒準備實作時，我往往會請 Claude 整理出一份實作計畫供我審查。這份計畫會特別聚焦在最有可能發生變動的部分，例如審查資料模型、型別介面或使用者體驗流程。這讓 Claude 能夠提早把那些我可能需要調整的部分浮上檯面。

提示詞範例：
「用 HTML 寫一份實作計畫，但開頭請先列出我最有可能會微調的決策：資料模型的變更、新的型別介面，以及任何面向使用者的改動。把那些機械式的重構埋在最下面就好，那部分我信任你。」

▍執行中階段：實作筆記（Implementation notes）
當我對計畫感到滿意後，我會開啟一個全新的對話 Session，並將所有產出物丟進提示詞中。例如，我可能會丟入規格文件和原型，然後要求 Agent 開始實作。

但事實是，無論你事前做了多少規劃，暗處總是潛伏著未知圈的未知。Agent 在工作時可能會發現，由於程式碼中出現了某個邊界狀況，它必須採取不同的策略。

我會要求 Claude Code 保持維護一個暫時性的 implementation-notes.md（或 .html）檔案，用來記錄它所做的決策，以便我們能從下一次的嘗試中學習。

提示詞範例：
「請維護一個 implementation-notes.md 檔案。如果你遇到迫使你偏離原定計畫的邊界狀況，請選擇相對保守的選項，將其記錄在『偏離事項』下，然後繼續進行。」

▍執行後階段：提案與說明（Pitches and explainers）
交付成果最重要的環節之一，就是獲得團隊的認同與批准。在最終文件中建立提案與說明產出物，可以達到以下效果：

加速理解：讓審查者從與你相同的未知起點出發，快速進入狀況。
加速批准：當專家們看到你已經把他們預期中的未知與常見失敗點都考慮進去時，會更容易放行。

提示詞範例：
「將原型、規格說明和實作筆記打包成一份單一文件，方便我直接丟到 Slack 爭取團隊認同。開頭請直接放上示範 GIF。」

▍執行後階段：隨堂測驗（Quizzes）
在經歷了一段漫長的協作後，Claude 完成的進度可能遠超我的想像。單純看程式碼的差異只能讓我產生表面上的理解，因為許多行為其實取決於既存的程式碼路徑。

在提供一堆上下文之後，要求 Claude 針對這次的變動對我進行隨堂測驗，能極大地幫助我理解到底發生了什麼事。我給自己訂下的規矩是：只有在我完美通過測驗後，才會合併這條分支。

提示詞範例：
「我想確保自己完全理解這次變更的所有細節。請為我生成一份 HTML 報告，說明這次變更的背景上下文、直覺邏輯、具體做了什麼等，並在最下方附上一個關於此次變更的隨堂測驗，我必須完全答對才行。」

▍這些字句如何匯聚成形：Fable 的發表
Fable 的發表影片完全是由 Claude Code 剪輯而成的。這對我來說是一個全新的領域，我絕非這方面的專家。

於是我從我已知的部分開始。我知道 Claude 可以透過撰寫程式碼來剪輯影片並進行語音轉文字，但我不敢確定它的精準度是否足夠。接著我請 Claude 向我解釋像 Whisper 這樣的語音轉文字技術是如何運作的，以及我是否能利用 ffmpeg 精準地剪掉諸如「呃、啊」等贅詞或過長的停頓。

我希望 Claude 創造一個能與我說話字句同步對齊的 UI，但我拿不准它是否辦得到，於是我請 Claude 先利用 Remotion 和轉錄文字製作一個影片原型，來測試這項科學實證是否可行。

最後，影片畫面看起來有點黯淡，我知道這是需要經過調色的結果，但我壓根不知道調色具體要怎麼做。我的第一直覺本來是想叫 Claude 直接做幾個不同版本讓我挑，但我馬上意識到，在調色這件事上，我根本不知道什麼叫做「好」。因此，我轉而要求 Claude 先教我關於調色的知識，藉此去發掘我的未知。

▍媒合地圖與疆域
當模型的實力變得越來越強大，只要方法正確，你能成就的事情就越多。當一個長週期的任務最後搞砸時，通常意味著你需要花更多時間去定義你的未知，或是去制定一個允許 Claude 在遇到未知時能靈活應變的實作計畫。

每一次的說明、腦力激盪、訪談、原型和參考資料，都是在事情變得積重難返、代價高昂之前，用最低廉的成本找出「你所不知道的事」的絕佳途徑。

所以，開啟你的下一個專案吧——就從請 Claude 幫你尋找你的未知開始。
