# 股票投資組合管理工具專案憲法（Constitution）

## 專案簡介
- 專案名稱：股票投資組合管理工具
- 短描述：提供投資人集中管理交易紀錄、持股成本、損益與資產配置的系統化工具。
- 目標使用者：有自行記錄交易需求的個人投資人、半主動投資者、需要跨帳戶追蹤資產的人。
- 核心價值：
  - 讓投資紀錄與損益計算具備一致性與可追溯性。
  - 讓投資人能以單一介面掌握持股、成本、報酬與配置狀況。

## 不可妥協原則（Non-negotiable Principles）
### 原則一：資料可追溯
- 說明：所有投資損益、持股與摘要資訊都必須可回推到原始交易紀錄。
- 必須遵守的規則：
  - 每筆交易都必須保留建立時間、交易日期、標的、數量、價格、費用與帳戶資訊。
  - 任何摘要數字都必須可由明細重建。
  - 交易異動需保留修改歷程或審計軌跡。
- 理由：金融資料若無法追溯，使用者無法信任結果，也不利於除錯與查核。[cite:24][cite:29]

### 原則二：金額與數量計算一致
- 說明：所有成本、損益與報酬率計算規則必須一致且文件化。
- 必須遵守的規則：
  - 明確定義成本法，例如平均成本法或 FIFO，且在 MVP 中只能有一套主要規則。
  - 手續費、交易稅、股利等項目是否計入成本或損益，必須有一致定義。
  - 金額欄位應採固定精度策略，不得使用未定義的浮點近似處理。
- 理由：同一組交易若在不同畫面出現不同結果，會直接破壞產品可信度。[cite:22][cite:24]

### 原則三：規格先於實作
- 說明：所有功能開發皆應先完成規格、釐清與計畫，再進入任務拆解與實作。
- 必須遵守的規則：
  - 新功能先更新 spec，再進行 clarify、plan、tasks。
  - 不可直接根據模糊需求生成程式碼。
  - 實作前應確認功能與憲法無衝突。
- 理由：Spec-Kit 的核心就是以規格驅動開發，避免 AI 或工程師過早進入實作。[cite:14][cite:16]

### 原則四：可驗證性優先
- 說明：每個核心功能都必須有清楚驗收條件與測試方式。
- 必須遵守的規則：
  - 所有損益計算邏輯都要有單元測試。
  - 交易建立、賣超阻擋、摘要一致性必須有整合測試。
  - 重要 UI 顯示值須能對應 API 與資料來源。
- 理由：Spec-Kit 強調從需求到任務都要能落到可驗證結果。[cite:9][cite:23]

## 領域與範圍（Scope & Boundaries）
- 本專案要處理的範圍：
  - 建立與維護股票交易紀錄。
  - 計算持股成本、未實現損益、已實現損益。
  - 顯示投資組合摘要與資產配置。
  - 支援多帳戶、多標的與基本查詢篩選。
- 本專案明確不處理的範圍（MVP）：
  - 不提供投資建議。
  - 不處理衍生性金融商品。
  - 不保證即時報價精準性。
  - 不在第一版支援複雜稅務申報與完整公司行為自動化。

## 開發與協作原則
- 規格驅動：所有功能需依序通過 Constitution → Specify → Clarify → Plan → Tasks → Implement。[cite:14][cite:16]
- 測試與品質：核心金流與損益邏輯必須有測試覆蓋；提交前需通過 lint、type check 與測試。
- 可維護性：模組命名清晰、欄位定義一致、任何商業規則都需文件化。
- 文件同步：若實作改變計算規則，必須同步更新 spec 與 plan。

## 安全與合規
- 資料安全：使用者資料、帳戶資訊與交易紀錄需受權限控管保護。
- 最小揭露：只顯示完成任務必要的敏感資料。
- 法規意識：產品內容屬投資紀錄管理工具，不應以介面或文案暗示投資顧問或保證獲利。
